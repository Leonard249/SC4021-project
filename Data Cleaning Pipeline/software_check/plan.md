# Software Check Plan

## Current Goal

Build a simple browser-based review tool for validating LLM-generated classifications.

- One post is shown at a time.
- The post's comments are shown underneath.
- Each review item shows the LLM label, confidence, reasoning, and evidence.
- Reviewer can `Accept` or `Reject`.
- If rejected, reviewer must choose an alternative label.
- There is an `Accept All` shortcut in the UI.

## Current Deployment Direction

- Frontend: Next.js app in `frontend/`
- Backend: FastAPI app in `backend/`
- Hosting target:
  - frontend on Vercel
  - backend on GCP VM

## Reviewer Queue Split

The copied combined dataset has been split into four reviewer DB files:

- `db_bryan.json` -> `Irrelevant`
- `db_ananya.json` -> `Neutral`
- `db_ryan.json` -> `Positive`
- `db_leonard.json` -> `Negative`

Reviewer metadata is stored in:

- `profiles.json`

The split is generated by:

- `scripts/build_reviewer_dbs.py`

## Data Contract

Each reviewer DB keeps:

- `llm`: original model prediction
- `review`: human decision state

This is important because the assigned queue bucket is only the starting prediction.
Example:

- item starts in Bryan's `Irrelevant` queue
- reviewer rejects it
- final human label becomes `Neutral`

We keep both values instead of overwriting the LLM output.
